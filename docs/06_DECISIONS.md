# 06. Принятые решения и разбор противоречий

## 1. Supabase является каноническим backend

В задании прямо указан Supabase. Поэтому:

- SQL-скрипт трактуется как основной backend-ориентир;
- `export_pocketbase.json` считается только дополнительным справочным источником;
- PocketBase SDK, коллекции и auth-flow не должны использоваться в реализации клиента.

## 2. Архитектурный файл из задания не приложен

В тексте задания есть ссылка на отдельное описание архитектуры, но сам файл не был приложен.

Поэтому в комплекте зафиксирована безопасная интерпретация требований:

- строгие слои `domain`, `data`, `presentation`;
- Clean Architecture;
- use case в `domain`;
- ViewModel + Compose в `presentation`.

## 3. Email regex: текст задания противоречит примеру

Формулировка говорит, что старший домен должен быть длиной больше двух символов, но пример использует `.ru`, где длина равна двум.

Чтобы не провалить сценарий с примером из задания, принимается такое решение:

```regex
^[a-z0-9]+@[a-z0-9]+\.[a-z]{2,}$
```

То есть TLD — от 2 символов.

Если преподаватель/проверка явно потребуют строго `> 2`, значение должно меняться централизованно в одном validator/константе, а не поиском по всему проекту.

## 4. Минимальная длина пароля

Задание не фиксирует длину пароля, но в экспортированном auth-описании PocketBase для поля `password` задан минимум 8 символов.

Поэтому в приложении принимается app-level правило:

- пароль должен быть не короче 8 символов.

Это безопасное и ожидаемое ограничение, не противоречащее UX.

## 5. OTP flow для восстановления пароля

Задание явно требует экран `OTP Verification`, хотя типовой Supabase recovery часто описывается через reset link/deeplink.

Принятое решение:

- сохранить именно OTP-сценарий, потому что он прямо требуется UX-заданием;
- recovery email в Supabase должен быть настроен на отправку токена/OTP;
- экран `Verification` проверяет именно recovery-код;
- только после успешной проверки доступен экран `Create New Password`.

Таким образом не ломается ни UX задания, ни общая модель Supabase Auth.

## 6. Регистрация и Confirm email

Supabase по умолчанию часто предполагает email confirmation. Но задание требует:

- успешная регистрация;
- после нее переход на `Sign In`.

Чтобы не встраивать отдельный экран подтверждения email, принимается практичное решение для учебного проекта:

- отключить `Confirm email` в настройках проекта Supabase;
- если sign-up создает сессию автоматически, сразу выполнить `signOut()` и вернуть пользователя на `Sign In`.

Это дает поведение, совпадающее с заданием.

## 7. Поле фото товара отсутствует в SQL

В SQL таблица `products` не содержит поля изображения, но:

- макет товара почти наверняка требует картинку;
- экспорт PocketBase содержит поле `photo` у продуктов.

Следовательно, это не временная опция, а обязательная доработка схемы.

Решение:

```sql
alter table public.products add column if not exists photo text;
```

Использовать то же имя поля, что и в `actions`/`profiles`, чтобы не усложнять data layer.

## 8. `favorite` vs `favourite`

Расхождение в названии не должно просачиваться в UI и domain.

Решение:

- в базе оставлять `favourite`, если проект уже строится на SQL-скрипте;
- в коде использовать нейтральные названия `Favorite`, `FavoritesRepository`, `FavoriteScreen`;
- преобразование имени таблицы скрыть внутри data layer.

## 9. Profile как отдельная таблица, а не расширение auth user

PocketBase исторически хранил многие пользовательские поля прямо в auth-коллекции. В Supabase и приложенном SQL эти данные вынесены в таблицу `profiles`.

Поэтому каноническая модель для клиента:

- `auth.users` — источник id/email/сессии;
- `public.profiles` — источник firstname/lastname/address/phone/photo.

## 10. Edit Profile — отдельный экран

Фраза из задания допускает как inline-редактирование, так и отдельный экран `Edit Profile`.

Для более чистой навигации и архитектуры принимается решение:

- реализовать `Edit Profile` как отдельный экран/route.

Это проще тестировать и не перегружает `Profile` экран лишними edit-state.

## 11. Штрихкод, а не QR

В задании прямо сказано «бар-код», поэтому:

- не использовать QR как основное представление;
- генерировать линейный штрихкод `Code128` из user id.

## 12. Что не нужно реализовывать сейчас

Не нужно расширять scope до cart/orders/payments/notifications, если этого не требует макет текущего задания.

Приоритет — полностью рабочие auth, home, catalog, favorite, profile, loyalty сценарии.
