# Комплект контекста для Codex — проект UP-01.03

Этот комплект подготовлен как стартовый контекст для реализации Android-приложения в **Android Studio** на **Kotlin + Jetpack Compose**.

В комплект входят:

- `AGENTS.md` — главный файл правил для Codex/агентов.
- `docs/01_SPECIFICATION.md` — продуктовая и функциональная спецификация.
- `docs/02_ARCHITECTURE.md` — техническая архитектура и структура проекта.
- `docs/03_SUPABASE_CONTRACT.md` — целевой контракт с Supabase, схема данных и RLS.
- `docs/04_IMPLEMENTATION_PLAN.md` — план реализации по этапам.

## Как использовать

1. Поместить `AGENTS.md` в корень репозитория.
2. Папку `docs/` положить в репозиторий, например в `docs/codex/`.
3. Перед генерацией кода читать файлы в таком порядке:
   1. `AGENTS.md`
   2. `docs/02_ARCHITECTURE.md`
   3. `docs/03_SUPABASE_CONTRACT.md`
   4. `docs/01_SPECIFICATION.md`
   5. `docs/04_IMPLEMENTATION_PLAN.md`

## Приоритет источников истины

1. Текст задания из DOCX.
2. Макет Figma как источник визуального соответствия.
3. SQL-скрипт как основной ориентир по целевой структуре PostgreSQL/Supabase.
4. `export_pocketbase.json` как вспомогательный источник для недостающих полей и бизнес-смыслов, но не как целевой backend.

## Что уже зафиксировано в комплекте

- Целевой backend: **Supabase**.
- Целевой стек клиента: **Kotlin + Jetpack Compose + Clean Architecture + MVVM/MVI-подобная подача состояния**.
- Минимальная версия Android: **11 (API 30)**.
- Валидация и бизнес-логика вынесены из UI в слой `domain`.
- Все сетевые операции обязаны иметь индикацию загрузки.
- Восстановление пароля сведено к OTP-сценарию, совместимому с Supabase Auth.
- Несоответствия между SQL и PocketBase вынесены в отдельный документ решений и отражены в контракте Supabase.

## Ключевые противоречия, которые уже обработаны

- В задании указан **Supabase**, но приложен экспорт **PocketBase**.
- В SQL таблица `products` не содержит поля изображения, хотя UI и PocketBase явно предполагают фото товара.
- В задании строка с email-валидацией противоречит примеру `name@domenname.ru`.
- В SQL используется имя таблицы `favourite`, а в JSON — `favorite`.
- В задании есть OTP-экран для reset password, тогда как стандартный поток Supabase часто описывается через reset link/deeplink.

Все эти расхождения разобраны в `docs/06_DECISIONS.md` и превращены в однозначные указания для реализации.
