# 01. Спецификация продукта и функциональности

## 1. Назначение приложения

Мобильное Android-приложение магазина обуви/спортивных товаров с авторизацией, восстановлением пароля, просмотром акций, каталогом, избранным, профилем пользователя и картой лояльности.

## 2. Платформа и стек

- Платформа: Android
- Минимальная версия: Android 11 / API 30
- Язык: Kotlin
- UI: Jetpack Compose
- Backend: Supabase

## 3. Scope MVP

В MVP входят:

- регистрация пользователя по email и паролю;
- вход по email и паролю;
- восстановление пароля через email + OTP verification + задание нового пароля;
- экран Home с акциями и товарными секциями;
- экран Catalog с загрузкой товаров с сервера;
- добавление/удаление избранного;
- экран Favorite с данными конкретного пользователя;
- экран Profile с загрузкой и обновлением профиля;
- смена фото профиля через камеру или галерею;
- генерация штрихкода из user id;
- экран Loyalty Card;
- Side Menu с logout;
- нижняя навигация.

## 4. Out of scope для текущего этапа

Хотя в SQL присутствуют таблицы `cart`, `orders`, `orders_items`, `payments`, `deliveries`, `notifications`, `order_status`, в рамках этого задания они не являются обязательной частью пользовательского сценария и не должны раздувать MVP.

Их можно подготовить как будущую базу расширения, но не использовать как условие готовности текущего приложения.

## 5. Пользовательские состояния

Приложение работает в двух основных состояниях:

### 5.1 Гость

Доступны экраны:

- `Sign In`
- `Register Account`
- `Forgot Password`
- `Verification`
- `Create New Password`

### 5.2 Авторизованный пользователь

Доступны экраны:

- `Home`
- `Catalog`
- `Favorite`
- `Profile`
- `Edit Profile`
- `Loyalty Card`
- `Side Menu`

## 6. Навигационный каркас

Рекомендуемые маршруты:

```text
auth/signin
auth/signup
auth/forgot
auth/otp?email={email}
auth/new-password?email={email}
main/home
main/catalog
main/favorite
main/profile
main/edit-profile
main/loyalty-card
```

Стартовое поведение:

- если активной сессии нет — старт с `Sign In`;
- если активная сессия есть — старт с `Home`.

## 7. Общие UX-правила

1. Для всех сетевых операций обязательно показывать загрузку.
2. Для действий отправки формы использовать блокировку повторного нажатия на время запроса.
3. Для пустых списков должны быть предусмотрены empty-state экраны/блоки.
4. Ошибки сервера должны приводиться к дружелюбному тексту.
5. Для всех экранов использовать theme из макета: цвета, типографика, отступы, скругления, иконки.

## 8. Функциональность по экранам

### 8.1 Register Account

**Назначение:** регистрация пользователя по email и паролю.

**Элементы:**
- поле `Email`
- поле `Password`
- toggle показа/скрытия пароля
- кнопка `Зарегистрироваться`
- текстовая ссылка/кнопка `Войти`

**Поведение:**
- email валидируется локально;
- пароль можно показать/скрыть;
- иконка глаза должна соответствовать текущему состоянию;
- при успешной регистрации выполняется переход на `Sign In`;
- по нажатию `Войти` выполняется переход на `Sign In`.

**Валидации:**
- email не пустой;
- email соответствует проектному regex (см. `06_DECISIONS.md`);
- пароль не пустой;
- пароль не короче 8 символов.

**Ошибки:**
- некорректный email — диалог;
- пустые/некорректные поля — диалог;
- серверная ошибка регистрации — диалог.

**Серверный сценарий:**
- `signUp(email, password)`;
- при успехе — переход на `Sign In`.

### 8.2 Sign In

**Назначение:** авторизация пользователя по email и паролю.

**Элементы:**
- поле `Email`
- поле `Password`
- toggle показа/скрытия пароля
- кнопка `Войти`
- ссылка `Восстановить`
- ссылка `Создать пользователя`

**Поведение:**
- проверка email;
- проверка на пустые поля;
- по нажатию `Восстановить` переход на `Forgot Password`;
- по нажатию `Создать пользователя` переход на `Register Account`;
- при успешном входе — переход в основной граф приложения.

**Ошибки:**
- некорректный email — диалог;
- пустые поля — диалог;
- неверный пароль / пользователь не найден — диалог.

### 8.3 Forgot Password

**Назначение:** инициировать восстановление пароля.

**Элементы:**
- поле `Email`
- кнопка `Отправить`
- кнопка/иконка `Назад`
- диалог успеха по макету

**Поведение:**
- при корректном email и успешной отправке запроса показать диалог;
- по нажатию на диалог перейти на `OTP Verification`;
- по `Назад` перейти на `Sign In`.

**Ошибки:**
- некорректный email — диалог;
- email не найден / серверная ошибка — диалог.

### 8.4 Verification / OTP Verification

**Назначение:** подтвердить код восстановления.

**Элементы:**
- 6 ячеек/слотов OTP-кода
- таймер `01:00`
- кнопка/ссылка повторного запроса кода
- кнопка подтверждения, если предусмотрена макетом

**Поведение:**
- пока таймер не истек, повторная отправка недоступна;
- после 60 секунд доступен повторный запрос кода;
- при неверном коде все ячейки окрашиваются в error-state;
- при успешной верификации переход на `Create New Password`.

**Примечание:**
- длину OTP в UI зафиксировать как 6 символов, если макет не противоречит этому.

### 8.5 Create New Password

**Назначение:** установить новый пароль после успешной верификации кода.

**Элементы:**
- поле `New Password`
- поле `Confirm Password`
- toggle показа/скрытия пароля
- кнопка `Сохранить`

**Поведение:**
- проверка совпадения паролей;
- после успешного изменения пароля — переход на `Sign In`.

**Валидации:**
- оба поля заполнены;
- пароль не короче 8 символов;
- пароли совпадают.

**Ошибки:**
- несовпадение паролей — диалог;
- слишком короткий пароль — диалог;
- серверная ошибка — диалог.

### 8.6 Home

**Назначение:** главный экран после авторизации.

**Содержимое:**
- верхняя часть по макету;
- блок акций из таблицы `actions`;
- секции товаров по макету;
- нижнее меню.

**Серверные данные:**
- акции: `actions`;
- рекомендуемые/бестселлеры: `products where is_best_seller = true`;
- при необходимости дополнительные подборки — из `products`.

**Состояния:**
- loading;
- content;
- empty promotions / empty products;
- error.

### 8.7 Product Card (переиспользуемый UI-компонент)

Обязательный отдельный компонент.

**Минимум должен поддерживать:**
- изображение товара;
- название;
- цену;
- бейдж/маркер best seller при необходимости;
- иконку/кнопку добавления в избранное;
- callback на открытие товара/карточки.

### 8.8 Bottom Navigation

Нижняя навигация должна соответствовать макету.

Рекомендуемый набор вкладок:
- Home
- Catalog
- Favorite
- Profile

Если порядок в Figma отличается — следовать Figma.

### 8.9 Catalog

**Назначение:** список товаров каталога.

**Данные:**
- `products`
- `categories`
- состояние избранного текущего пользователя

**Функции:**
- загрузка товаров с сервера;
- отображение карточек товаров;
- добавление/удаление в избранное;
- при наличии макета — фильтрация по категориям.

**Состояния:**
- loading;
- content;
- empty;
- error.

### 8.10 Side Menu

Состав и визуал — по макету.

**Функционально обязательно:**
- пункт `Выйти`;
- logout выполняет серверную деавторизацию и локальную очистку сессии;
- после успеха переход на `Sign In`.

### 8.11 Favorite

**Назначение:** список избранных товаров текущего пользователя.

**Данные:**
- таблица `favourite` + связанные товары из `products`.

**Функции:**
- загрузка избранного с сервера;
- отображение только товаров текущего пользователя;
- удаление из избранного;
- корректное empty-state состояние.

### 8.12 Profile

**Назначение:** отображение профиля пользователя.

**Данные:**
- `profiles` + текущий `auth user id`.

**Функции:**
- загрузка данных профиля с сервера;
- переход к `Edit Profile`;
- смена фото профиля;
- генерация штрихкода из id пользователя;
- переход на `Loyalty Card` по нажатию на штрихкод.

**Поля профиля:**
- имя
- фамилия
- адрес
- телефон
- email (из auth/session, read-only либо отображаемый отдельно)
- фото профиля

### 8.13 Edit Profile

Экран/режим редактирования профиля по макету `Edit Profile`.

**Функции:**
- изменение текстовых полей профиля;
- изменение фотографии;
- сохранение на сервере;
- отображение прогресса сохранения.

**Изменение фото:**
- по нажатию на кнопку `Изменить фото профиля` показывать диалог/лист выбора:
  - камера
  - галерея

### 8.14 Loyalty Card

**Назначение:** экран карты лояльности.

**Содержимое:**
- крупный штрихкод;
- при необходимости имя пользователя и вспомогательный текст по макету.

**Источник данных:**
- user id из активной сессии;
- профиль пользователя для подписи, если это есть в макете.

## 9. Общие правила валидации

### 9.1 Email

Использовать единый валидатор в `domain`.

Проектное правило:

```regex
^[a-z0-9]+@[a-z0-9]+\.[a-z]{2,}$
```

Причина выбора зафиксирована в `docs/06_DECISIONS.md`.

### 9.2 Password

- минимум 8 символов;
- пустое значение недопустимо;
- совпадение паролей обязательно на экране создания нового пароля.

### 9.3 OTP

- только цифры;
- длина 6 символов;
- неверный код переводит все ячейки в error-state.

## 10. Нефункциональные требования

- приложение не должно падать при пустом ответе сервера;
- все ошибки Supabase должны быть обработаны безопасно;
- повторный вход после logout не должен использовать старую сессию;
- UI должен оставаться отзывчивым во время загрузок;
- изображения должны загружаться асинхронно и кешироваться клиентом.
