# AGENTS.md

Ты работаешь над Android-приложением по учебному проекту UP-01.03.

Проект должен быть реализован в **Android Studio**, на **Kotlin**, UI — только **Jetpack Compose**.

## 1. Главная цель

Собрать работоспособное Android-приложение по макету Figma и тексту задания с backend на **Supabase**, строго соблюдая разделение на слои `domain`, `data`, `presentation`.

## 2. Обязательные ограничения

1. Минимальная версия Android: **11 / API 30**.
2. UI — только **Jetpack Compose**. Не возвращаться к XML-разметке, кроме технически необходимых Android-манифестных и файловых конфигураций.
3. Архитектура должна сохранять независимость экранов и слоев:
   - `domain` не зависит ни от `data`, ни от `presentation`;
   - `data` может зависеть только от `domain` и инфраструктурных библиотек;
   - `presentation` может зависеть только от `domain` и UI/Android API.
4. Любая валидация и бизнес-правила должны жить не в composable-функциях, а в `domain`.
5. Любая сетевая операция должна показывать индикатор загрузки.
6. Все тексты ошибок, где это требует задание, показывать через **диалоговое окно**, а не только через inline-error.
7. Секреты Supabase не хранить в репозитории. Использовать `local.properties`, Gradle properties или другой некоммитящийся способ инъекции `SUPABASE_URL` и `SUPABASE_ANON_KEY`.
8. Никогда не использовать `service_role` ключ внутри мобильного клиента.

## 3. Источники истины

Используй приоритет:

1. `docs/06_DECISIONS.md`
2. `docs/03_SUPABASE_CONTRACT.md`
3. `docs/02_ARCHITECTURE.md`
4. `docs/01_SPECIFICATION.md`
5. Figma — как визуальный источник, если он доступен
6. SQL / PocketBase-материалы только в том виде, как они уже интерпретированы внутри документов выше

Если визуальная деталь расходится со спецификацией, а логика не затронута, приоритет у Figma.
Если вопрос касается backend, моделей, ролей доступа, навигации и поведения — приоритет у документов этого комплекта.

## 4. Что делать в первую очередь

Перед добавлением крупного объема кода:

1. Настроить базовый проект Android.
2. Подключить Compose, Navigation, DI, Supabase client.
3. Поднять theme и базовые UI-компоненты.
4. Реализовать auth-graph.
5. Только после этого переходить к main-graph и интеграции с серверными данными.

## 5. Рекомендуемый стек реализации

Допускается использовать следующие технологии без изменения общей архитектурной модели:

- `Navigation Compose`
- `ViewModel`
- `Kotlin Coroutines` + `Flow`
- `kotlinx.serialization`
- DI: `Hilt` или `Koin` (предпочтительно `Hilt`, если нет внешних ограничений)
- Supabase Kotlin client (`auth`, `postgrest`, `storage`)
- `Coil` для загрузки изображений
- `CameraX` и/или `ActivityResultContracts`
- библиотеку генерации штрихкода на базе `ZXing`/совместимого решения

Не добавляй зависимости «на всякий случай». Каждая зависимость должна закрывать конкретную задачу.

## 6. Правила по структуре проекта

Используй feature-oriented разбиение внутри слоев.

Пример допустимой структуры:

```text
app/
  src/main/java/.../
    data/
      auth/
      profile/
      products/
      favorites/
      promotions/
      media/
    domain/
      auth/
      profile/
      products/
      favorites/
      common/
    presentation/
      auth/
        signin/
        signup/
        forgot/
        otp/
        newpassword/
      home/
      catalog/
      favorite/
      profile/
      loyalty/
      components/
      navigation/
      theme/
    di/
    core/
```

`core/` допустим только для нейтральных утилит, которые не содержат бизнес-логики и не ломают правило зависимости слоев.

## 7. Правила по экранным моделям

Для каждого экрана использовать:

- `UiState`
- `UiEvent`/`Intent`
- `UiEffect` для одноразовых действий (навигация, диалог, toast/snackbar)
- отдельный `ViewModel`

Не смешивай состояние нескольких экранов в одном ViewModel, если это не контейнерный shell-экран.

## 8. Backend-правила

1. Целевой backend — **Supabase**, а не PocketBase.
2. Для аутентификации использовать Supabase Auth.
3. Для профиля — таблицу `public.profiles`.
4. Для каталога — `public.products` и `public.categories`.
5. Для акций — `public.actions`.
6. Для избранного — `public.favourite` (несмотря на расхождение в написании).
7. Для файлов использовать Supabase Storage.
8. Если поля не хватает в SQL, сначала применять миграцию из `docs/03_SUPABASE_CONTRACT.md`, а не хардкодить временный client-side workaround.

## 9. UX-правила

1. Все состояния загрузки должны быть видны пользователю.
2. Ошибки валидации, которые прямо указаны в задании, показывать через диалог.
3. Повторный запрос OTP разблокируется по истечении таймера 01:00.
4. При неверном OTP все ячейки кода должны перейти в error-state.
5. Переключение видимости пароля должно менять и состояние поля, и иконку.
6. Карточка товара должна быть отдельным переиспользуемым компонентом.
7. Штрихкод строить из `auth user id` и открывать отдельный экран `Loyalty Card` по нажатию.

## 10. Что нельзя делать

- Не переносить логику запросов прямо в composable.
- Не обращаться к Supabase из UI напрямую.
- Не смешивать DTO, domain-модели и UI-модели в один класс.
- Не дублировать email-валидацию в трех местах разными regex.
- Не хранить временно важные решения только в комментариях к коду; фиксируй их в понятных именах, use case и централизованных validator/mapper.
- Не реализовывать tables/orders/cart/payments как часть MVP, если это не требуется текущим макетом.

## 11. Definition of Done

Задача считается выполненной, если одновременно соблюдены все пункты:

1. Экран соответствует макету по структуре и визуальной логике.
2. Навигация соответствует заданию.
3. Валидации и состояния ошибок работают.
4. Серверные запросы реально подключены к Supabase.
5. Есть загрузка/ошибка/успех для сетевых сценариев.
6. Логика разнесена по слоям корректно.
7. Не нарушены документы `docs/02_ARCHITECTURE.md` и `docs/03_SUPABASE_CONTRACT.md`.
8. Код не содержит моков там, где в задании требуется реальный сервер.

## 12. Если возникает неоднозначность

Не придумывай новый backend-поток с нуля.

Сначала:

1. смотри `docs/06_DECISIONS.md`,
2. затем `docs/03_SUPABASE_CONTRACT.md`,
3. затем `docs/01_SPECIFICATION.md`.

Если после этого остаются варианты, выбирай тот, который:

- не ломает Clean Architecture,
- укладывается в Supabase Auth/PostgREST/Storage,
- минимизирует количество нестандартной серверной логики,
- не расходится с UX из задания.
